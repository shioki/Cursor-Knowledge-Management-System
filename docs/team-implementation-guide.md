# 🏢 チーム導入ガイド：Cursor知見管理システム

## 📋 概要

個人での成功事例をもとに、チーム全体でCursor知見管理システムを導入するためのガイドです。

---

## 🎯 チーム導入の目標

- **属人化解消**: 個人の知見をチーム全体で共有
- **オンボーディング効率化**: 新メンバーの学習コスト削減
- **品質標準化**: 一貫したコード品質とベストプラクティス
- **開発スピード向上**: チーム全体の生産性向上

---

## 📅 3段階導入プロセス

### Phase 1: 個人試用期間（2週間）

#### 目標
- 各メンバーが個人レベルでシステムを理解
- カスタマイズポイントの発見
- 効果の実感

#### 実施内容
- [ ] 全メンバーが個人プロジェクトで導入
- [ ] 基本的な使い方をマスター
- [ ] 1人1つ以上の成功事例を作成
- [ ] 課題・改善点を記録

#### 成果物
- 個人での使用感レポート
- プロジェクト固有のカスタマイズ案
- チーム共有すべき知見リスト

### Phase 2: チーム標準化期間（3週間）

#### 目標
- チーム共通のルール・フォーマット策定
- 試験的なチーム運用開始
- 運用プロセスの確立

#### 実施内容
- [ ] チーム標準テンプレートの作成
- [ ] ファイル命名規則・タグ付けルールの統一
- [ ] 記録タイミング・責任者の明確化
- [ ] 週次振り返り会の設定

#### Week 1: 標準化準備
```markdown
- チーム会議でPhase 1の結果共有
- 共通テンプレート案の作成
- ルール案の策定
```

#### Week 2: 試験運用
```markdown
- 標準テンプレートでの運用開始
```

#### Week 3: ルール確定
```markdown
- 運用ルールの最終決定
- チーム全体での合意形成
- 本格運用準備
```

### Phase 3: 本格運用・継続改善（継続）

#### 目標
- 安定したチーム運用の確立
- 継続的な改善サイクルの構築
- 新しいベストプラクティスの発見

#### 実施内容
- [ ] 月次振り返り会の定期開催
- [ ] 四半期での効果測定
- [ ] 新メンバー向けオンボーディング資料の作成
- [ ] 他チームへの知見共有

```markdown
## Cursorへの指示

### 基本方針
- このプロジェクトはチーム開発です
- `.cursor/`内のファイルからチームの過去知見を参照してください
- 新しい知見は必ず適切なファイルに記録してください
- 設計判断は**必ず理由と決定者**を明記してください

### チーム開発ガイドライン
1. **コンテキスト確認**: `.cursor/context.md`でプロジェクト背景・制約を確認
2. **既存パターン活用**: `.cursor/patterns.md`でチーム標準パターンを確認  
3. **重要な解決策記録**: 30分以上の問題解決は`.cursor/debug-log.md`に記録
4. **改善提案**: 失敗例や改善案は`.cursor/improvements.md`に記録
5. **設計判断記録**: アーキテクチャ決定は`.cursor/knowledge.md`に記録

## 記録ルール
- **即座記録**: 問題解決後、直ちに記録
- **責任者明記**: 記録者・決定者を明確に記載
- **レビュー必須**: 重要な知見は他メンバーのレビューを受ける
- **定期整理**: 月1回、チーム全体で知見整理を実施

## 活用方法
```

# 新しい知見の記録
解決後、該当ファイルに記録 + Slackでチーム共有

## 🔄 チーム運用プロセス

### 日常運用

#### 個人レベル
```markdown
1. 問題発生時
   - まずチーム知見を検索: @.cursor/debug-log.md
   - 新しい問題なら調査・解決後に記録
   - 重要な知見はSlackでチーム共有

2. 新機能開発時  
   - 設計前にパターン確認: @.cursor/patterns.md
   - 新パターン発見時は即座に記録
   - 設計判断は理由と共に記録

3. コードレビュー時
   - 既存パターンとの整合性確認
   - 新しいベストプラクティス発見時は記録
```

#### チームレベル
```markdown
週次ミーティング（15分）:
- 今週追加された重要な知見の共有
- 来週注意すべき技術的課題の確認

月次振り返り（1時間）:
- 知見の整理・統合
- 運用プロセスの改善検討
- チーム外への知見共有準備
```

### 品質管理

#### 知見のレビューフロー
```markdown
1. 個人記録
   ↓
2. チーム内レビュー（重要度に応じて）
   ↓  
3. 承認・統合
   ↓
4. チーム標準化
```

#### レビュー観点
- [ ] **正確性**: 技術的に正しいか
- [ ] **再現性**: 他メンバーが実行可能か  
- [ ] **網羅性**: 必要な情報が含まれているか
- [ ] **検索性**: 後で見つけやすいか

## 🚀 成功のポイント

### 導入成功要因
1. **トップダウン支援**: マネージャーの積極的な推進
2. **ボトムアップ参加**: メンバー全員の主体的な参加  
3. **継続的改善**: 定期的な振り返りとプロセス改善
4. **適度な強制力**: ルール化による継続性確保

### よくある失敗パターンと対策

#### 失敗パターン1: 記録が属人化
**原因**: 特定の人だけが記録している
**対策**: 
- 記録責任の分散（持ち回り制）
- 記録しやすい環境整備
- 記録の価値を実感できる仕組み作り

#### 失敗パターン2: 情報が散在
**原因**: 記録ルールが不統一
**対策**:
- 明確な記録基準の設定
- テンプレートの統一
- 定期的な整理プロセス

#### 失敗パターン3: 形骸化
**原因**: 効果が実感できない、面倒
**対策**:
- 小さな成功体験の積み重ね
- 定期的な効果測定・共有
- プロセスの継続的な簡素化
