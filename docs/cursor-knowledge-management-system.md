# Cursor知見管理システム 導入ガイド

## 📋 概要

Cursor AIを活用した効率的な開発知見管理システムです。プロジェクトが大きくなるにつれて、過去の試行錯誤や設計判断をAIに効率的に伝え、知見を活用する仕組みが重要になります。

### 解決する課題
- 同じ問題について何度も説明が必要になる
- 過去の設計判断の理由がAIに伝わっていない
- プロジェクト固有の制約や要件を毎回説明する手間
- デバッグで得た知見が散逸しやすい

### 期待される効果
- **開発効率向上**: 同じ説明の繰り返しが80%減少
- **デバッグ時間短縮**: 問題解決時間が40%短縮
- **チーム連携強化**: 新メンバーのオンボーディング時間半減
- **コード品質向上**: 一貫性向上とアンチパターン再発防止

## 📁 ファイル構成と役割

```
project-root/
├── CURSOR.md                   # 🎯 メインエントリーポイント
└── .cursor/
    ├── context.md              # 📋 プロジェクト背景・制約
    ├── patterns.md             # 🔧 頻用パターン・コマンド集
    ├── debug-log.md            # 🐛 重要なデバッグ記録
    ├── improvements.md         # 📈 改善履歴・教訓
    ├── knowledge.md            # 💡 技術的知見・設計判断
    └── debug/
        ├── sessions/           # 📝 セッション別ログ
        ├── temp-logs/          # 🗂️ 一時ファイル
        └── archive/            # 📦 解決済み問題保管
```

### 各ファイルの詳細説明

| ファイル | 目的 | 更新頻度 | 記録内容 |
|---------|------|----------|----------|
| **CURSOR.md** | Cursorへの指示・プロジェクト概要 | 初期設定時 | AI活用方針、基本情報 |
| **context.md** | プロジェクト背景・制約 | 月1回 | ビジネス要件、技術制約、設計原則 |
| **patterns.md** | 頻用パターン・コマンド | 随時 | よく使うコマンド、実装テンプレート |
| **debug-log.md** | デバッグ記録 | 問題解決時 | 30分以上の調査、重要な問題解決 |
| **improvements.md** | 改善履歴・教訓 | プロセス改善時 | 失敗例、学んだ教訓、改善アクション |
| **knowledge.md** | 技術知見・設計判断 | 設計決定時 | アーキテクチャ決定、技術選定理由 |

## 📝 テンプレート集

### CURSOR.md テンプレート

```markdown
# [プロジェクト名] AI開発知見管理システム

## プロジェクト概要
[プロジェクトの目的と概要を記述]

## Cursorへの指示

### 基本方針
- このプロジェクトの過去知見を`.cursor/`ディレクトリ内のファイルから参照してください
- 新しい知見や解決策は適切なファイルに記録してください
- 設計判断は必ず理由と共に記録してください

### 開発ガイドライン
1. **コンテキスト活用**: `.cursor/context.md`でプロジェクト背景を確認
2. **パターン参照**: `.cursor/patterns.md`で既存パターンを確認
3. **デバッグ記録**: 重要な問題解決は`.cursor/debug-log.md`に記録
4. **改善履歴**: 失敗から学んだ教訓を`.cursor/improvements.md`に蓄積

## 知見更新タイミング
- 新しい設計パターン発見時
- 重要なバグ解決時（30分以上要したもの）
- 効率化手法発見時
- アンチパターン発見時

## 定期メンテナンス
- 月1回の知見整理
- 重複情報の統合
- 古い情報のアーカイブ化
```

## 🔄 運用方法

### 日常的な運用

#### 1. 問題解決時のワークフロー
```
1. 問題発生 → 既存知見を検索（@.cursor/debug-log.md）
2. 新しい問題の場合 → 調査開始
3. 解決したら即座に記録（debug-log.md）
4. 重要な学びは knowledge.md にも記録
```

#### 2. 新機能開発時のワークフロー
```
1. 設計前に既存パターン確認（@.cursor/patterns.md）
2. 設計判断は理由と共に記録（knowledge.md）
3. 実装中に新パターン発見 → patterns.md に追加
4. 完了後、改善点があれば improvements.md に記録
```

#### 3. Cursor活用時のベストプラクティス
```
# 効果的な質問方法
@.cursor/context.md この制約の下で、[具体的な質問]

# 複数ファイル参照
@.cursor/patterns.md @.cursor/knowledge.md を参考に、[質問]

# 段階的な改善
選択範囲 → Ctrl+I → AIに改善提案依頼
```

### 定期メンテナンス（月1回推奨）

#### チェックリスト
- [ ] 重複情報の統合
- [ ] 古い情報の更新またはアーカイブ
- [ ] 新しく学んだパターンの patterns.md への追加
- [ ] 成功・失敗事例の improvements.md への記録
- [ ] ファイルサイズが大きくなった場合の分割検討

#### アーカイブの運用
```powershell
# 解決済み問題のアーカイブ（例）
Move-Item ".cursor\debug\temp-logs\2024-01-*" ".cursor\debug\archive\"
```

## 🔍 トラブルシューティング

### よくある問題と解決策

#### Q1: ファイルが多すぎて管理が困難
**A1**: 
- 月1回の定期整理を実施
- 使用頻度の低いファイルをアーカイブ
- テーマ別にディレクトリを分割

#### Q2: 記録を忘れがち
**A2**:
- 問題解決と同時に記録する習慣化
- Cursorのコメント機能を活用してリマインダー設定
- チームでの相互リマインド

#### Q3: 検索性が悪い
**A3**:
- 一貫したタグ付けルールの設定
- ファイル内の見出し構造を統一
- 定期的なキーワード整理

#### Q4: 情報が古くなる
**A4**:
- 作成日・更新日を記録
- 月1回の情報鮮度チェック
- 古い情報の適切なアーカイブ
